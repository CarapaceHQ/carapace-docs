# System Architecture

## Overview

The Carapace platform provides a **behavior-based trust layer for AI agents interacting with APIs, services, and infrastructure**.

The system collects behavioral telemetry from agents, analyzes that telemetry for suspicious patterns, computes risk scores, and enables services to make automated trust decisions.

Carapace is designed to operate in **local-only environments, self-hosted deployments, or federated trust networks**.

The architecture separates responsibilities into several independent components to allow flexible deployment and horizontal scaling.

# Core Components

## Telemetry Collection

Telemetry is generated by systems interacting with AI agents.

This may include:

* application middleware
* API gateways
* LLM security middleware
* agent runtime environments
* browser extensions
* reverse proxies

Telemetry collectors transform observations into **Risk Events** that follow the standardized event schema.

These events contain:

* agent identity
* behavioral metadata
* network context
* content signals
* economic signals
* timestamps

These events form the foundation of behavioral analysis across the platform.

---

## Event Ingestion Gateway

The Event Ingestion Gateway receives telemetry events and validates them against the Carapace event schema.

Responsibilities include:

* schema validation
* authentication of telemetry sources
* rate limiting
* event normalization
* batching and buffering

Validated events are forwarded to the streaming pipeline for processing.

---

## Streaming Pipeline

The streaming pipeline enables real-time behavioral analysis.

Typical responsibilities include:

* event transport
* stream partitioning
* buffering and backpressure management
* ordering guarantees
* distributed processing

Example technologies may include:

* Kafka
* Redpanda
* cloud streaming services

Streaming architecture allows the platform to scale to high volumes of agent activity.

---

## Detection Engine

The Detection Engine evaluates behavioral telemetry and identifies suspicious patterns.

Detection methods may include:

* rule-based detection
* threshold detection
* temporal pattern analysis
* correlation logic
* signature detection
* anomaly detection

Detection outputs include:

* risk flags
* detection records
* scoring inputs

These outputs are forwarded to the Risk Scoring Engine.

---

## Risk Scoring Engine

The Risk Scoring Engine converts detections into a normalized **risk score between 0 and 100**.

Risk scores combine multiple categories of signals:

* network risk
* behavioral risk
* content risk
* economic risk
* reputation adjustments

The scoring engine also produces:

* risk flags
* score history
* confidence metrics

These scores represent the likelihood that an agent is behaving maliciously.

---

## Trust Graph

The Trust Graph models relationships between entities interacting within the system.

Nodes may represent:

* agents
* organizations
* services
* infrastructure
* economic identities

Edges represent behavioral interactions and shared infrastructure.

The graph enables detection of coordinated activity such as:

* agent farms
* botnet infrastructure
* fraud rings
* shared malicious networks

Graph intelligence improves detection accuracy and reputation modeling.

---

## Policy Engine

The Policy Engine translates trust scores into automated enforcement decisions.

Possible actions include:

* allow request
* challenge agent
* rate limit activity
* require verification
* quarantine agent
* block traffic

Policies may be configured by organizations based on their risk tolerance.

---

## Trust API

The Trust API allows external systems to query the trust layer.

Example use cases include:

* checking an agent risk score
* retrieving triggered risk flags
* inspecting behavioral history
* integrating trust decisions into application logic

The API enables services to incorporate behavioral trust decisions into their workflows.

---

## Operator Dashboard

The dashboard provides visibility into the platform's behavioral intelligence.

Features may include:

* agent profile pages
* risk score history
* event timelines
* triggered detections
* trust graph visualizations
* policy configuration

The dashboard enables investigation and operational management of the trust layer.

# Deployment Models

Carapace supports multiple deployment models depending on organizational needs.

## Local Deployment

All telemetry collection, detection, and scoring occurs within the organization's infrastructure.

No telemetry leaves the environment.

## Self-Hosted Platform

Organizations deploy the Carapace platform within private cloud or on-premise infrastructure.

This allows full control over telemetry and analysis.

## Hybrid Trust Network

Organizations analyze telemetry locally while sharing **derived trust signals** with a federated trust network.

This enables cross-organization intelligence without exposing raw telemetry.

## Managed Service

A managed Carapace platform may provide hosted scoring infrastructure and trust APIs.

Organizations send telemetry to the hosted platform and query trust decisions via API.

---

# Design Principles

The architecture follows several guiding principles.

Local-first deployment support

Organizations should be able to run the full platform locally.

Privacy preservation

Raw telemetry remains under the control of the organization.

Derived signals may be shared across the trust network.

Explainability

All detections and scores should be explainable to operators.

Extensibility

New detection rules, telemetry signals, and scoring models should be easily added.

Scalability

Streaming pipelines allow the platform to process large volumes of telemetry events.
